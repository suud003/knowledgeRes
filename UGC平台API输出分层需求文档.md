# UGC 平台 API 输出分层需求文档

## 重点特性

**对 UGC 平台三类开放 API 建立分级管理体系，在文档和代码 Core 文件夹中进行定性标注，明确各层级的风险等级、标注方式和后续处理方案。**

## 需求背景

当前 UGC 平台脚本编辑功能允许玩家编写代码并运行在 DS 机上，与其他玩法在同一环境运行。平台已对外开放三种来源不同、安全等级各异的 API 接口，但缺乏统一的分级标注和风险管控机制。参考和平项目的经验，开放接口后需花费大量人力添加保护措施（如限制绿洲玩法中某些物品的发放），表面看似"裸开放"的背后实际有严格的安全措施。为避免当前平台陷入同样的安全风险，需建立完整的 API 分级管理体系。

## 需求目的

**建立 API 三级分层标注体系**，在文档侧（VS Code 插件提示、API 文档）和代码侧（Core 文件夹源码注解）同步落地，让开发者在使用 API 时能清楚感知风险等级，同时为后续安全团队审核、发行团队沟通提供规范化依据。

## 需求内容

### 一、API 三级分层定义

#### 1. API 分层总览

| 层级 | 来源 | 风险等级 | 稳定性承诺 | 标注标识 |
|---|---|---|---|---|
| **L1 — 安全接口** | 经过时间验证的可视化编程 API，经 OOP 面向对象改造 |  **低风险** | 长期稳定，不会轻易变动 | `@ApiLevel(STABLE)` |
| **L2 — 受控接口** |  人工筛选和测试后包装的 API，约 1000 个 |  **中风险** | 尽量确保不变动，但可能微调 | `@ApiLevel(CONTROLLED)` |
| **L3 — Beta 接口** |  完全未经筛查，已在半开放平台暴露的大量接口 |  **高风险** | 随时可能删除或修改 | `@ApiLevel(BETA)` |

#### 2. 各层级详细说明

##### L1 — 安全接口（STABLE）

- **功能描述**：已经过长期可视化编程验证的 API，经过 OOP 面向对象改造后形成安全接口
- **安全保障**：经过比较完善的安全审计和功能测试，用户可放心使用
- **变动策略**：原则上保持接口不变；
- **用户感知**：文档中不标注任何风险提示，作为推荐使用的标准接口

##### L2 — 受控接口（CONTROLLED）

- **功能描述**：经人工筛选并包装的 API（约 1000 个），投入较大人力进行筛选和测试
- **安全保障**：经过人工审核和测试，尽量确保调用安全可靠
- **变动策略**：尽量不变动，但后续版本可能进行微调；变动前会提前通知
- **用户感知**：文档中标注 `⚠️ 受控接口 — 后续版本可能微调`

##### L3 — Beta 接口（BETA）

- **功能描述**：已在半开放平台暴露的大量接口，未经人工筛查
- **安全保障**：**不提供安全保障**，存在较大安全风险（ps：【*需与安全团队确认单独处理方案】）
- **变动策略**：**随时可能删除或修改** — L3 接口变更时，通过平台公告和编辑器内提示通知受影响的开发者
- **用户感知**：文档中标注 `Beta 接口 — 此接口处于 Beta 阶段，不保证稳定性，随时可能关闭或变更，使用前请充分评估风险`

### 二、文档侧标注方案

#### 1. API 文档标注

- **每个 API 条目**须在文档开头明确标注所属层级和风险等级徽章
- **L3 Beta 接口**须额外增加醒目的风险说明框：

```
⚠️ BETA 接口警告
此接口处于 Beta 阶段，未经过安全审核和稳定性验证。
风险提示：
  1. 接口可能在后续版本中被删除或修改
  2. 使用该接口开发的游戏功能可能因接口变更而失效
  3. 该接口可能存在安全风险，请勿用于敏感操作
建议：仅用于实验性功能开发，核心玩法请使用 L1/L2 接口
```

#### 2. VS Code 插件标注

| 标注位置 | L1 安全接口 | L2 受控接口 | L3 Beta 接口 |
|---|---|---|---|
| **代码补全列表** | 普通图标 | ⚠️ 黄色标记 | 红色标记 + `[BETA]` 前缀 |
| **悬浮提示** | 接口说明 | 接口说明 + 受控接口提示 | 接口说明 + Beta 警告全文 |
| **诊断信息** | 无 | 无 | `Warning` 级别提示："您正在使用 Beta 接口，此接口可能随时变更" |
| **代码补全排序** | 优先 | 正常 | 最后 |

### 三、代码侧标注方案（Core 文件夹）

#### 1. 标注注解定义

在 Core 文件夹中定义统一的 API 层级注解/装饰器：

```lua
-- API 层级枚举
ApiLevel = {
    STABLE = 1,       -- L1: 安全接口，经 OOP 改造的可视化编程 API
    CONTROLLED = 2,   -- L2: 受控接口，人工筛选包装的 API
    BETA = 3           -- L3: Beta 接口，半开放平台暴露 API
}
```

#### 2. Core 文件夹标注规范

| 标注项 | 说明 | 示例 |
|---|---|---|
| **文件级标注** | 每个 API 模块文件头部声明该文件中 API 的默认层级 | `-- @module_api_level STABLE` |
| **函数级标注** | 每个对外暴露函数须标注层级 | `-- @api_level BETA` |
| **废弃标注** | L3 接口如被计划移除，增加废弃标注 | `-- @deprecated 计划在 V4.x 移除` |

#### 3. 标注示例

```lua
---------------------------------------------------
-- @module: PlayerInventory
-- @module_api_level: STABLE
-- @description: 玩家背包系统接口
---------------------------------------------------

--- 获取玩家背包物品列表
-- @api_level STABLE
-- @return table 物品列表
function PlayerInventory:GetItemList()
    -- ...
end

--- 直接修改物品属性
-- @api_level CONTROLLED
-- @warning 受控接口，后续版本可能微调
function PlayerInventory:ModifyItemProperty(itemId, property, value)
    -- ...
end

--- 发送自定义物品到玩家（未筛查的半开放接口）
-- @api_level BETA
-- @warning Beta 接口，随时可能删除或修改
function PlayerInventory:SendCustomItem(itemId, count)
    -- ...
end
```
