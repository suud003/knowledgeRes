# 打字数据识别规则详解

## 识别算法

### 第一层：关键词匹配

以下关键词命中时，对应内容大概率为工作相关：

**项目/任务类**
- 项目代号、产品名称（需用户首次补充）
- "需求"、"排期"、"版本"、"迭代"、"里程碑"
- "提测"、"上线"、"灰度"、"发布"、"回滚"
- "CR"、"Code Review"、"MR"、"PR"

**技术类**
- 代码相关：函数名、类名、文件路径、错误堆栈
- "接口"、"API"、"联调"、"Mock"
- "Bug"、"Crash"、"性能"、"内存"、"CPU"
- "数据库"、"缓存"、"队列"、"服务"

**协作类**
- "会议"、"评审"、"对齐"、"同步"、"拉齐"
- "邮件"、"文档"、"Wiki"
- "@某人"（同事名字/花名）

**进度类**
- "完成"、"搞定"、"已提交"、"已合入"
- "进行中"、"调试中"、"排查中"
- "计划"、"明天"、"下一步"、"todo"

### 第二层：上下文推断

当单条数据无法直接判断时，结合上下文：
- 前后相邻的工作内容可作为辅助判断
- 与已知项目名相近的简称/缩写需关联
- 技术讨论中穿插的非正式表达视为工作内容

### 第三层：排除规则

以下内容排除出工作报告：
- 纯表情符号
- 外卖/购物/游戏等生活内容
- 与任何项目无关的闲聊
- 重复/无意义的输入

## 分类策略

### 自动分类优先级

1. 精确匹配：内容中直接包含项目名
2. 模块匹配：内容涉及某项目已知的模块/组件
3. 上下文关联：与同一时间段的其他已分类内容关联
4. 用户确认：无法自动分类时，展示给用户选择

### 进度状态推断

| 关键词 | 推断状态 |
|--------|---------|
| 完成、搞定、已解决、提交、合入、上线 | ✅ 已完成 |
| 在做、调试、联调中、开发中、排查 | 🔄 进行中 |
| 明天、计划、准备、下一步、后续 | 📋 计划中 |
| 卡住、阻塞、等待、依赖 | ⚠️ 阻塞中 |

## 数据清洗

### 内容精炼规则

- 口语化表达 → 书面化描述（如 "那个bug搞了半天终于搞定了" → "修复XXX问题"）
- 碎片化内容 → 合并为完整描述
- 去除重复提及的同一工作内容
- 保留关键数据和结论

### 示例

**原始打字数据：**
```
早上到了 先看了下邮件
今天要把那个脚本编辑器的安全评估搞完
中午吃了个沙拉 还行
下午和老王对了下引擎接口开放的方案 确定了风险等级标记的方式
哦对了 顺便帮测试那边看了个crash 是空指针问题 已经修了
明天要继续搞接口配置的整理
```

**识别结果：**
```
工作相关（脚本编辑器项目）：
- [进行中] 脚本编辑器安全评估
- [完成] 与老王讨论引擎接口开放方案，确定风险等级标记方式
- [计划] 继续整理接口配置

工作相关（其他）：
- [完成] 修复测试反馈的空指针Crash问题

已过滤：
- "早上到了 先看了下邮件" → 通用事项，不纳入
- "中午吃了个沙拉 还行" → 非工作内容
```
