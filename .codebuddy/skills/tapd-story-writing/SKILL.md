# TAPD 需求内容编写规范

## Overview

PUBG Mobile 项目的需求单内容编写标准。提供需求单的标准结构模板、编写原则、总单子单同步规则、格式规范和修改检查清单。用于指导需求描述（description 字段）的内容组织和格式化，确保需求单质量一致。

## 一、TAPD 需求单标准结构模板

**一句话加粗概括本需求的核心交付**

1-2句陈述事实和问题。历史需求用超链接引用，例如：

## 二、需求单编写原则

1. **边界清晰**：在「需求目的」或「需求内容」开头明确本单处理范围和排除项
2. **排除项不删除**：不在范围内的内容单列说明，附上负责的关联需求单链接
3. **关联双保障**：TAPD 建立 story-to-story 关联 + 描述中附超链接
4. **背景不重复**：需求目的不要复述背景，只写核心动作
5. **历史需求用链接**：不要内联大段旧需求描述，用超链接引用

## 三、总单与子单的同步规则

1. **背景统一**：总单和子单的「需求背景」保持一致，改总单必须同步子单
2. **总单是索引**：按编号列出所有改造项，每项标注负责方和关键结论
3. **子单是详情**：引用总单背景，展开本模块的功能点、参数表、流程图
4. **改动联动**：总单某项调整 → 对应子单 + 关联需求单同步更新

## 四、格式规范

1. **加粗**（`**文字**`）：核心动作、关键结论、范围声明
2. **表格**：对比类信息（现状vs调整后）、参数、字段、指标一律用表格
3. **Mermaid**：业务流程用 Mermaid 流程图，不用文字描述流程
4. **引用链接**：历史需求、关联需求单一律用 `[标题](url)` 超链接
5. **编号层级**：大章节用 `##`，子章节用 `###`，功能点用 `#### 1. 2. 3.`
6. TAPD 会过滤内联 HTML，不要用 `<span style>` 做颜色，统一用 Markdown 加粗

## 五、需求单修改检查清单

每次修改前对照：

- [ ] 重点特性是否一句话概括清楚？
- [ ] 背景是否与总单一致？历史需求是否用链接引用？
- [ ] 需求目的是否只写核心动作、不重复背景？
- [ ] 每个功能点是否包含：触发条件 + 校验逻辑 + 未达标处理？
- [ ] 是否有现状vs调整后对比表？
- [ ] 关键参数是否附定标依据？
- [ ] 处理范围和排除项是否明确？
- [ ] 关联需求单是否建立 TAPD 关联 + 文本链接？
- [ ] 总单/子单是否同步更新？
- [ ] 风险是否附应对建议？
- [ ] 收益指标是否有基线和预期目标？

## 六、TAPD 描述样式限制

1. TAPD 需求描述 **不支持可靠的红字、彩色字体或内联 HTML 样式**，不要承诺"红色标注"可稳定生效。
2. 如用户要求重点标注，统一使用以下可稳定方案：
   - `**加粗**`
   - `【重点补充】`
   - `【必须修改】`
   - 独立小节标题，如 `### 重点补充`
3. 如果必须表达"红字同等强调效果"，应明确告知用户：TAPD 内建议改为文本高亮风格；如确需彩色，改在 Word/飞书文档/图片评审稿中处理。

## 七、TAPD 链接写法规范

1. TAPD 对 Markdown 链接的渲染和回显不稳定，写入后可能只保留标题文本，URL 不一定稳定展示。
2. 当需求中必须保证链接肉眼可见时，优先使用：
   - 一行标题
   - 下一行裸链 URL
3. 推荐写法：

```
**关联历史需求**：
【【优先二轮适配】【V420】【WOW开局流程】作品大厅三期——⑧ 增加作品大厅筛选列表】
https://tapd.woa.com/tapd_fe/20360302/story/detail/1020360302126335981
```

## 八、增量修改原则

1. 用户要求"尽量增量修改"时，优先保留原有章节、编号和正文结构。
2. 新增内容应尽量插入到对应章节内部，不要默认统一追加到文末。
3. 若总单和子单都存在对应章节，修改时需同步更新，避免口径不一致。
