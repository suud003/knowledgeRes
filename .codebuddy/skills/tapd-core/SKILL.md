## 语义映射表（查询意图 → 字段对照）

### 人员相关语义

用户在查询缺陷时常用不同说法，需映射到正确的查询字段：

| 用户说法 | 实际含义 | 查询字段 | 示例值 |
|---------|---------|----------|-------|
| "需要我处理的 bug" / "未处理的 bug" / "待处理的 bug" | 当前处理人是我 | `current_owner` | `tianyishao;` |
| "我负责的 bug" / "我是负责策划的 bug" | 负责策划是我 | `custom_field_9` | `tianyishao` |
| "我的 bug"（模糊） | 两者都查，合并去重 | `current_owner` + `custom_field_9` | 分两次查询 |

> ⚠️ `current_owner` 的值末尾带分号（如 `tianyishao;`），而 `custom_field_9` 不带分号。

### 版本相关语义

用户提到版本时，需完成两步转换：**简写→完整版本号** + **选择正确的版本字段**。

**第一步：版本号转换**

用户可能用简写（如 `430`）或完整版本号（如 `4.3.0`），需通过 `references/version-schedule.md` 映射为完整版本号。

| 用户输入 | 转换结果 | 说明 |
|---------|---------|------|
| `430` / `4.3` / `4.3.0` | `4.3.0` | 简写自动补全 |
| "当前版本" / "这个版本" | 按「版本自动判断」规则 | 见下方章节 |
| "外网版本" / "线上版本" | 按「版本自动判断」规则 | 见下方章节 |

**第二步：选择正确的版本字段**

| 查询对象 | 版本字段 | ⚠️ 注意 |
|---------|---------|---------|
| 需求（Story） | `version` | 如 `"version": "4.3.0"` |
| 缺陷（Bug） | `version_report` | 如 `"version_report": "4.3.0"` |

> ⚠️ **不可混用**：查缺陷时必须用 `version_report`，查需求时必须用 `version`。这是最常见的错误之一。

### 查询场景默认参数

用户的自然语言查询通常隐含了确定的参数组合，以下为常见场景的默认值：

| 场景关键词 | 默认状态（status） | 默认人员字段 | 说明 |
|-----------|-------------------|-------------|------|
| "缺陷情况" / "bug情况" / "有多少bug" | `new\|in_progress\|reopened` | `current_owner`（处理人=我） | 分析类查询默认查待处理的 |
| "待处理bug" / "未处理bug" | `new\|in_progress\|reopened\|audit_reject` | `current_owner`（处理人=我） | 包含审核驳回 |
| "我负责的bug" | `new\|in_progress\|reopened` | `custom_field_9`（负责策划=我） | 角色是负责策划 |

> ⚠️ **"我的XX版本缺陷"中的"我" = 处理人（`current_owner`）**，不是负责策划。只有明确说"我负责的"才用 `custom_field_9`。

## MCP 动态调用链路

当前 MCP 配置为动态工具加载模式（X-Tools-Set: lookup_tapd_tool），标准调用链路如下：

1. **获取工具参数 schema**：调用 `lookup_tool_param_schema`，传入 `tool_name`
2. **代理执行工具**：调用 `proxy_execute_tool`，传入 `tool_name` 和 `tool_args`

无需提前调用 `lookup_tapd_tool` 进行语义搜索（当已知工具名时可跳过）。

## 版本自动判断

根据当前日期与 `references/version-schedule.md` 中的**后台更新**节点时间对比，自动判断当前开发版本和外网版本。

### 判断规则

1. **当前外网版本**：后台更新日期 ≤ 今天的最新版本
2. **当前开发版本**：找到**后台更新日期 > 今天**的最早版本（记为 V），然后判断：
   - 若今天距离 V 的后台更新日期 **≥ 2 周**（14天），当前开发版本 = V
   - 若今天距离 V 的后台更新日期 **< 2 周**（14天），当前开发版本 = V 的下一个版本（即 V+1）

### 判断示例

假设今天是 `2026-03-06`：
- 4.3.0 的后台更新是 `2026-03-11`（> 今天，但距今仅 5 天 < 14 天）→ 4.3.0 即将更新，开发重心已切到下个版本
- → **当前开发版本 = 4.4.0**
- 4.2.0 的后台更新是 `2026-01-06`（≤ 今天）→ **当前外网版本 = 4.2.0**

假设今天是 `2026-04-01`：
- 4.4.0 的后台更新是 `2026-05-11`（> 今天，距今 40 天 ≥ 14 天）→ **当前开发版本 = 4.4.0**
- 4.3.0 的后台更新是 `2026-03-11`（≤ 今天）→ **当前外网版本 = 4.3.0**

### 应用场景

| 用户说法 | 对应版本 | 判断依据 |
|---------|---------|---------|
| "当前版本" / "这个版本" / "新版" | 当前开发版本 | 后台更新日期 > 今天的最早版本 |
| "线上版本" / "外网版本" / "上个版本" | 当前外网版本 | 开发版本的前一个版本 |
| "下个版本" / "下版" | 开发版本 +1 | 开发版本的后一个版本 |
| 具体版本号或简写（如 `430`、`4.3.0`） | 直接匹配 | 精确匹配版本号或简写 |

> ⚠️ 涉及版本字段（需求 `version`、缺陷 `version_report`）时，用户可能使用简写（如 `430`），需自动转换为完整版本号（如 `4.3.0`）。

## 人员 ID 自动匹配

涉及人员字段（处理人 `owner`/`current_owner`、负责策划 `custom_field_79`/`custom_field_9`、前台开发 `custom_field_82`、后台开发 `custom_field_78`）时，用户可能提供以下非标准格式：

| 用户输入类型 | 示例 | 匹配方式 |
|------------|------|---------|
| 中文全名 | 邵天熠 | 精确匹配中文名 → `tianyishao` |
| 拼音全拼 | shiqingce | 模糊匹配英文ID中的拼音部分 |
| 英文简写 | kenny | 前缀/子串匹配英文ID → `tianyishao` |
| 英文全ID | tianyishao | 直接使用 |

### 匹配规则

1. **精确匹配优先**：先在 `references/people-mapping.md` 中精确匹配中文名或英文ID
2. **模糊匹配**：无精确结果时，按英文ID前缀/子串匹配，或按中文姓氏+名字部分匹配
3. **多结果处理**：如匹配到多个人，列出候选列表让用户选择
4. **无结果处理**：如完全无法匹配，主动询问用户提供准确的企业微信英文ID
5. **格式修正**：匹配到 ID 后，根据字段要求自动添加末尾分号（`owner`/`current_owner` 字段需要 `;` 后缀）

> ⚠️ **所有人员字段必须使用企业微信英文ID**，不接受中文名直接写入 TAPD。执行任何涉及人员的操作前，必须先完成 ID 匹配确认。

## TAPD 文本能力限制

> 以下限制供各子 Skill 复用，在涉及 TAPD 内容编辑时需主动告知用户。

- 不支持可靠红字/彩色字
- Markdown 链接回显不稳定
- 重点内容建议用 `**加粗** + 【重点补充】`
- 关键链接建议使用"标题 + 裸链 URL"
