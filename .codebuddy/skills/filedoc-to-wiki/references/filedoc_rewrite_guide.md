# FileDoc 改写规范

## 基础引用

本规范基于 `folderdoc-to-wiki/references/rewrite_style_guide.md` 的通用改写风格，并针对 **FileDoc（单文件深度解析文档）** 的特点做以下扩展。

> **核心区别**：FolderDoc 精简版追求"一页概览"，FileDoc 追求"深度参考"。FileDoc 的内容量通常是 FolderDoc 精简版的 3~10 倍，改写时应**保留详细深度**，仅做格式适配和冗余清理。

---

## FileDoc 与 FolderDoc 的改写差异

| 维度 | FolderDoc 精简版 | FileDoc 详细版 |
|------|-----------------|---------------|
| 内容深度 | 一句话职责 + 核心表格 | 完整职责描述 + 所有属性/函数详表 |
| 函数列表 | 仅核心函数（10~20个） | 全部函数，按职责分类 |
| 数据结构 | 核心属性表 | 完整成员变量表（含类型、默认值、说明） |
| 依赖关系 | 不包含 | 完整依赖链（继承、引用、被引用） |
| 流程图 | 关键流程 1~2 个 | 所有核心流程 mermaid 图 |
| 设计说明 | 不包含 | 保留完整的设计决策说明 |

---

## FileDoc 改写规则

### 1. 文件概览 → 职责 + 文件属性

将 FileDoc 中的"文件概览"表格拆分为两个章节：

**原始格式（FileDoc 中的表格）：**
```markdown
| 属性 | 值 |
|------|-----|
| 文件路径 | xxx/xxx.lua |
| 文件大小 | 约 XXX 行 |
| 继承关系 | 继承自 XXX |
| 主要职责 | ... |
| 运行端 | DS/Client/Both |
```

**改写为：**
```markdown
# 职责

{主要职责的完整描述，可多句话}

# 文件属性

| 属性 | 值 |
|------|-----|
| 文件路径 | `xxx/xxx.lua` |
| 文件大小 | 约 XXX 行 |
| 继承关系 | 继承自 `XXX` |
| 运行端 | DS/Client/Both |
```

### 2. 依赖关系 → 完整保留

FileDoc 中的"依赖关系"章节是 FolderDoc 精简版没有的增量信息，**必须完整保留**。

格式适配规则：
- 继承链保留为列表或表格
- 引用的外部类保留完整列表
- 被引用关系（如有）保留

### 3. 数据结构 → 完整表格保留

FileDoc 中的"数据结构/成员变量"通常是大型表格，**完整保留所有行**。

格式适配规则：
- 表格列标准化为：`| 属性名 | 类型 | 默认值 | 说明 |`（根据原文列调整）
- 如果原文按分类分组（如"基础属性"、"战斗属性"、"网络同步属性"），保留分组结构
- 网络同步属性（RepNotify）单独成节或在表格中标注

### 4. 函数列表 → 保留分类结构

FileDoc 的函数列表通常按职责分类（初始化、生命周期、战斗、网络同步等），**保留分类结构和所有函数条目**。

格式适配规则：
- 每个分类作为二级标题 `##`
- 函数表格标准化为：`| 函数名 | 说明 |` 或 `| 函数名 | 参数 | 说明 |`
- 如原文对关键函数有详细说明段落，保留为函数表格下方的补充说明

### 5. mermaid 流程图 → 完整保留

FileDoc 中的所有 mermaid 流程图**原样保留**（iWiki 支持 mermaid 渲染）。

格式适配规则：
- 保留 ` ```mermaid ` 代码块格式
- 流程图前的上下文说明文字保留
- 如有多个流程图，每个保留独立的二级标题

### 6. 关键设计说明 → 完整保留

"关键设计说明"章节是 FileDoc 的核心增量价值，**必须完整保留**。

包含但不限于：
- 架构设计决策
- 为什么这样实现（而不是那样）
- 已知的限制和注意事项
- 与其他系统的交互约定

## FileDoc 输出模板（按类型）

### Actor 实体类（如 MonsterBaseActor）

```markdown
# 职责

{完整的职责描述}

# 文件属性

| 属性 | 值 |
|------|-----|
| 文件路径 | `xxx.lua` |
| 文件大小 | 约 XXX 行 |
| 继承关系 | 继承自 `XXX` |
| 运行端 | DS/Client/Both |

# 依赖关系

## 继承链
- `当前类` → `父类` → `祖父类` → ...

## 引用的外部类
| 类名 | 用途 |
|------|------|
| ... | ... |

## 基础属性
| 属性名 | 类型 | 默认值 | 说明 |

| ... | ... | ... | ... |

## 战斗属性（如有）
（同上格式）

| 属性名 | 同步方向 | 回调函数 | 说明 |
|--------|---------|---------|------|
| ... | ... | ... | ... |

# 函数分类总览

## 初始化与生命周期
| 函数名 | 说明 |
|--------|------|
| ... | ... |

## 战斗相关（如有）
| 函数名 | 说明 |
|--------|------|
| ... | ... |

## 网络同步
| 函数名 | 说明 |
|--------|------|
| ... | ... |

## 交互逻辑
| 函数名 | 说明 |
|--------|------|
| ... | ... |

（按原文分类继续...）

` ``mermaid

` ``

（同上格式）

{完整保留原文的设计说明内容}
```

### 其他类型

对于非 Actor 类型的 FileDoc（如 Feature、SubSystem、Handler 等），参照基础改写规范中对应类型的模板结构，但同样遵循 FileDoc 的"保留详细深度"原则——即所有表格、流程图、设计说明均完整保留。

---

## 必须移除/简化的内容

即使 FileDoc 追求详细，以下内容仍应清理：

1. **引导性段落**：如"本文档将详细介绍..."、"在阅读本文之前..."等 → 直接移除
2. **重复的总结段落**：如章节末尾对前文的重复总结 → 移除
3. **过长的代码片段**（超过 15 行）→ 精简为关键片段或用表格替代
4. **目录索引**（FileDoc 自动生成的 TOC） → 移除（iWiki 有自己的目录功能）
5. **文档生成元信息**（如"本文档由 AI 生成"） → 移除

---

## 格式适配要点（iWiki 兼容）

1. 标题层级：`#` 为文档一级标题，`##` 为章节，`###` 为子章节，最多 4 级
2. 表格：确保每列对齐，表头与分隔线完整
3. mermaid：使用 ` ```mermaid ` 代码块，确保语法正确
4. 代码块：使用 ` ```lua ` 标注语言
5. 链接：iWiki 内部链接使用 `[文档名](docid)` 格式（如有需要）
6. 加粗/斜体：正常使用 `**粗体**` 和 `*斜体*`
